# v0.2.5 增长与国际化能力增强

## 版本概述

v0.2.5 聚焦“可发现、可回看、可反馈、可扩展”四个方向，补齐公告触达、生成结果沉淀、用户反馈收集、SEO 曝光、英文版本和 API Key 引导等基础能力，降低新用户上手门槛，增强老用户留存与运营迭代效率。

## 当前实现基线（已审查）

本版本规划已基于当前代码实现重新校准，而不是按假设重画界面。

**已审查的现有页面与壳层**：
- `src/App.vue`：应用壳层使用 `AppLayout` + `RouterView` 命名插槽。
- `src/components/layout/AppLayout.vue`：当前为左侧菜单栏 + 条件配置栏 + 主内容区的三栏结构。
- `src/components/layout/AppMenu.vue`：当前菜单包含首页、拆解脚本、脚本生成、我的收藏、设置、登录/个人中心。
- `src/views/HomePage.vue`：当前首页为 Hero + 两张功能卡片 + 即将推出区域。
- `src/views/AnalyzePage.vue` / `src/views/CreatePage.vue`：当前仅承载 `RightPanel`，实际配置区在命名路由里由 `LeftPanel` 提供。
- `src/components/LeftPanel.vue` / `src/components/RightPanel.vue`：当前分析与生成流程已共用“左配置、右结果”的工作台结构。
- `src/components/CreateModePanel.vue`：当前生成页已是单面板信息流，不再是旧版 Tabs 结构。
- `src/views/SettingsPage.vue`：当前仅有单列 API Key 表单，尚无语言设置、反馈区和分区式信息架构。

**审查结论**：
- v0.2.5 需要在现有三栏工作台和侧边菜单体系上增量扩展。
- 不应再按“顶部导航 + 独立横向导航栏”的思路规划核心工作区。
- 历史记录、公告、设置重构都应贴合现有组件边界与路由组织方式。

## 需求 1：增加更新公告功能

### 1.1 背景与目标

**当前问题**：
- 当前系统缺少正式的版本更新公告入口，用户无法快速感知新功能上线。
- 首页目前只有 Hero、功能卡片和“即将推出”区域，没有承载版本动态的固定位置。
- 产品侧缺少“首次访问提醒 + 已读状态留存”的标准机制。

**目标**：
- 在不破坏现有首页和工作台结构的前提下，增加公告横幅与公告弹窗。
- 支持当前版本公告展示与已读管理。
- 支持后续多语言公告扩展。

### 1.2 需求详情

**功能要求**：
1. 首页支持顶部公告横幅，展示当前版本摘要与“查看详情”。
2. 用户首次进入系统时，如存在未读公告，展示公告弹窗或抽屉。
3. 公告内容支持标题、版本号、摘要、正文、发布时间、状态管理。
4. 已登录用户记录公告已读状态；未登录用户使用浏览器本地缓存记录。
5. 公告文案支持中英文版本。

**非功能要求**：
- 公告展示不能阻塞首页或工作台加载。
- 公告入口需要与当前首页卡片式布局风格兼容。
- 公告浮层在桌面端与移动端均可正常展示。

### 1.3 验收标准

- [ ] 首页支持公告横幅展示当前版本摘要。
- [ ] 首次进入时会看到未读公告浮层，关闭后不重复弹出。
- [ ] 已登录用户公告已读状态可同步。
- [ ] 公告内容支持中英文文案配置。

## 需求 2：完善历史记录功能，记录每一次脚本生成结果

### 2.1 背景与目标

**当前问题**：
- 当前已有“我的收藏”页面，但没有完整的“每次生成都自动沉淀”的历史记录能力。
- 分析和生成的结果主要存在 `useVideoAnalysis` 的全局状态里，离开流程后缺少系统留档。
- 用户无法在现有侧边菜单中回看每次产出的结果，也无法区分成功/失败任务。

**目标**：
- 新增独立“历史记录”入口，与当前侧边菜单结构保持一致。
- 自动记录视频拆解、从零创作、参考生成三类结果。
- 支持筛选、查看详情、继续生成、收藏、导出等动作。

### 2.2 需求详情

**功能要求**：
1. 对以下模式的每次结果自动留档：视频拆解、从零创作、参考生成。
2. 每条历史记录至少包含：模式、标题、生成时间、模型、输入摘要、输出脚本内容、分镜结构、Token 消耗、状态。
3. 历史记录页支持按模式、时间、状态筛选，并支持关键词搜索。
4. 用户可从历史记录执行“查看详情 / 继续生成 / 收藏 / 导出”。
5. 失败任务也要保留失败状态与错误信息。
6. 历史记录入口需接入当前左侧菜单，不影响现有收藏页。

**非功能要求**：
- 历史详情展示不依赖已过期的 `oss://` 临时资源。
- 历史记录页优先适配当前桌面端壳层布局。
- 数据量增长后需支持分页或增量加载。

### 2.3 验收标准

- [ ] 三种模式的结果均会自动进入历史记录。
- [ ] 历史记录支持筛选、搜索和分页浏览。
- [ ] 用户可从历史记录重新查看结构化脚本结果。
- [ ] 历史记录支持再次收藏和导出。
- [ ] 失败任务会保留错误原因。

## 需求 3：增加功能反馈区，支持用户提交需求反馈

### 3.1 背景与目标

**当前问题**：
- 当前设置页只有 API Key 输入，没有产品反馈入口。
- 用户反馈只能通过产品外渠道收集，无法沉淀为结构化数据。

**目标**：
- 在当前设置页基础上扩展出“功能反馈”分区。
- 支持用户提交需求、问题和体验建议。
- 为后续版本排期提供可回溯的数据基础。

### 3.2 需求详情

**功能要求**：
1. 在设置页中增加“功能反馈”区域或分区卡片。
2. 表单支持反馈类型、标题、详细描述、期望结果、联系方式等字段。
3. 用户提交后给出明确成功提示，并展示提交时间。
4. 已登录用户可查看自己提交过的反馈记录与处理状态。
5. 数据结构支持区分“待处理 / 评估中 / 已采纳 / 已完成 / 已关闭”。

**非功能要求**：
- 反馈区重构需与设置页未来的语言设置、API Key 引导共存。
- 表单字段尽量简洁，控制提交成本。
- 需具备基础防刷策略。

### 3.3 验收标准

- [ ] 设置页可进入功能反馈区。
- [ ] 用户可成功提交需求或问题反馈。
- [ ] 反馈记录可在个人侧查看提交状态。
- [ ] 反馈数据可按类型和状态区分。

## 需求 4：完善 SEO 支持，提升外部检索与收录能力

### 4.1 背景与目标

**当前问题**：
- 当前首页内容较轻，主要是功能入口卡片，静态可抓取文本不足。
- 路由标题仅通过 `router.beforeEach` 写入简单页面标题，缺少更完整的 SEO 元信息。
- 公开页面缺少 `sitemap.xml`、`robots.txt`、结构化数据和英文公开落地内容。

**目标**：
- 在不改动现有首页核心结构的前提下，为首页补充可抓取的静态内容区。
- 建立基础 SEO 设施：标题、描述、OG、站点地图、robots、结构化数据。
- 为中英文公开页面提供基础可搜性。

### 4.2 需求详情

**功能要求**：
1. 为首页、登录页、设置页等公开页面配置独立 title / description / canonical。
2. 在当前首页卡片区域下方补充功能介绍、使用场景、FAQ、版本亮点等内容区。
3. 生成 `sitemap.xml` 与 `robots.txt`。
4. 首页注入 `SoftwareApplication` / `FAQPage` 等结构化数据。
5. 支持 Open Graph / Twitter Card 元信息。
6. 英文公开页至少具备对应英文标题、描述和核心介绍文案。

**非功能要求**：
- SEO 增强不影响现有 SPA 主流程。
- 元信息配置应支持按语言版本切换。
- 首页新增静态内容需要与现有 Hero 和功能卡片风格兼容。

### 4.3 验收标准

- [ ] 首页与主要公开页面具备独立 SEO 元信息。
- [ ] 站点可访问 `sitemap.xml` 与 `robots.txt`。
- [ ] 首页包含可抓取的功能介绍与 FAQ 区块。
- [ ] 首页具备基础结构化数据。
- [ ] 英文公开页或英文首页元信息可被识别。

## 需求 5：增加英文版本支持，并区分界面语言与 AI 回复语言

### 5.1 背景与目标

**当前问题**：
- 当前所有页面文案都是中文写死，尚未建立统一国际化层。
- 界面语言与 AI 输出语言无法分开配置。
- 设置页当前结构过于简单，尚无承载双语言设置的空间。

**目标**：
- 支持中文、英文双语界面。
- 在设置页中分别配置“界面语言”和“AI 回复语言”。
- 让首页、菜单、空状态、表单、公告、反馈等都具备语言适配能力。

### 5.2 需求详情

**功能要求**：
1. 支持至少 `zh-CN` 与 `en-US` 两种界面语言。
2. 设置页新增“界面语言”和“AI 回复语言”两个独立配置项。
3. 切换界面语言后，菜单、按钮、表单、空状态、错误提示同步切换。
4. AI 回复语言需写入分析/生成请求上下文，影响输出脚本语言。
5. 用户语言偏好在本地持久化；已登录用户可同步到个人资料。
6. 公告、反馈、SEO 文案支持按语言版本输出。

**非功能要求**：
- 默认语言保持中文，避免影响现有用户。
- 切换界面语言后无需刷新页面即可生效。
- 国际化改造需优先覆盖当前已实现的主路径页面。

### 5.3 验收标准

- [ ] 界面语言与 AI 回复语言可独立配置。
- [ ] 核心页面和公共组件支持中英文切换。
- [ ] AI 输出内容能根据设置切换中英文。
- [ ] 语言偏好支持本地持久化，登录后可同步。

## 需求 6：完善 API Key 配置引导，引导用户访问阿里云百炼申请 Key

### 6.1 背景与目标

**当前问题**：
- 当前设置页仅提供输入框和链接，对新用户不够友好。
- 分析页和生成页虽然会提示“请先配置 API Key”，但缺少更完整的引导闭环。

**目标**：
- 在现有设置页基础上升级为“申请引导 + 填写 + 校验”的完整流程。
- 从分析页、生成页空状态快速引导到设置页完成配置。
- 降低首次使用门槛，提升配置完成率。

### 6.2 需求详情

**功能要求**：
1. 设置页展示清晰的三步引导：前往阿里云百炼控制台、创建/查看 API Key、复制回系统粘贴保存。
2. 为新用户提供一键跳转阿里云百炼控制台入口。
3. 已填写 API Key 时支持脱敏展示、修改和清空。
4. 保存后提供基础可用性校验结果与错误提示。
5. 在分析页、生成页缺少 API Key 时，提供去设置页的直达入口。

**非功能要求**：
- API Key 仍仅保存在浏览器本地，不写入业务数据库。
- 引导文案需兼容中英文。
- 不改变现有环境变量兜底逻辑。

### 6.3 验收标准

- [ ] 设置页具备可执行的 API Key 申请与配置指引。
- [ ] 用户可一键跳转至阿里云百炼申请页。
- [ ] 已配置 Key 支持脱敏展示和重新编辑。
- [ ] 保存后可获得成功或失败校验反馈。
- [ ] 分析页与生成页在缺少 Key 时提供直达设置入口。
