### 审查结论

- 当前应用主壳层是“左菜单 + 条件配置栏 + 主内容区”，不是顶部导航站点结构。
- 分析页与生成页已经复用同一套工作台框架，不应在 v0.2.5 规划里假设为两套独立页面布局。
- 生成页当前已是单面板信息流，不再是旧版 Tabs；v0.2.5 需要在此基础上追加历史、语言和引导能力。
- 设置页仍是最轻量的单列表单，是 v0.2.5 最明确的重构点之一。

## 版本规划

```
v0.2.5-alpha  ──▶  v0.2.5-beta  ──▶  v0.2.5-stable
     │                   │                    │
     │                   │                    └── 回归验证 + 文案校对 + 发布公告
     │                   │
     │                   └── 设置页重构 / SEO / 双语 / 反馈联调
     │
     └── 审查现状 + 数据结构 + 历史记录 / 公告基础能力
```

### 里程碑定义

- **alpha**：完成代码审查、历史记录数据设计、公告数据设计、菜单入口与页面骨架
- **beta**：完成设置页重构、API Key 引导、语言配置、反馈区和 SEO 主体内容
- **stable**：完成中英文文案校对、首页 SEO 细节、全链路回归与版本公告发布

## 关键实现原则

- 历史记录、公告、反馈和设置重构必须基于当前三栏工作台和侧边菜单进行增量扩展。
- 不新增与现有 `AppLayout` 冲突的全局导航结构。
- 新页面与新组件默认延续 Composition API + `<script setup lang="ts">`。
- 设计文档必须明确区分“当前已实现结构”和“本版本增量变化”。

## 风险与依赖

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 文档规划继续偏离现有代码结构 | 高 | 实施前先审查相关页面与组件，文档记录审查文件清单 |
| 历史记录依赖 `oss://` 地址过期 | 高 | 核心展示以结果文本、脚本结构和输入摘要为主 |
| 设置页信息密度快速膨胀 | 中 | 使用分区卡片或标签分区，控制单屏复杂度 |
| 多语言覆盖不完整导致体验割裂 | 高 | 先覆盖侧边菜单、首页、设置页、空状态和主流程提示 |
| SEO 在 SPA 中收益有限 | 中 | 优先做首页静态内容、元信息、结构化数据和 sitemap |
